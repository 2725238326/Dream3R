# 平台部分重新定位：修改方案

## 核心定位（三层价值）

平台不只是"管几个模型"，它的价值分三层：

1. **辅助实验**（当前）：统一调度、统一格式、统一对比，让架构消融和多模型对照实验在可控条件下进行
2. **快速部署与对比**（近期方向）：让前馈式 3R 模型这一类模型可以快速部署、一键运行、即时对比或直接使用
3. **封装为 API 服务下游**（长期方向）：把模型能力封装成标准 API，供 SLAM 初始化、AR 建模、机器人场景理解等应用场景直接调用

## PPT 需要改的页面

### Slide 13 — 为什么需要一个聚合平台

**当前顶部 claim：**
> 六个模型各有各的环境和格式，手动对齐效率太低、结果难复现。

**改为：**
> 3R 模型缺少统一的部署、对比和调用基础设施，实验效率和应用落地都受限。

**正文四个方框内容调整：**

| 位置 | 当前 | 改为 |
|------|------|------|
| 框 1 | 环境差异：每个模型都有自己的运行环境和依赖 | **部署成本高**：每个模型各有环境和依赖，部署一次耗时长 |
| 框 2 | 格式差异：输入、输出和结果目录结构各不相同 | **对比困难**：输入输出格式不统一，横向比较需大量手动对齐 |
| 框 3 | 工具限制：Nerfstudio 面向 NeRF；商业产品闭源 | **现有工具不适用**：Nerfstudio 面向 NeRF，商业产品闭源 |
| 框 4 | 复现与平台需求：依赖版本影响复现；需统一调度、格式和对比 | **下游无法调用**：模型能力停留在脚本级，缺少标准接口供应用集成 |

**底部 claim 改为：**
> 统一部署、统一对比、标准化输出与 API 封装

## 讲稿对应修改

以下是讲稿中需要同步调整的段落（已写好替换文字）：

### Slide 13 讲稿

```
谢谢。接下来我讲平台部分。

前面的架构验证需要消融、对照、多模型对比，这些实验前提是条件统一。但问题不止于此——3R 模型目前连快速部署都做不到，更别说把能力封装出去给别的应用用。

每个模型各有环境和依赖，部署一次就要配半天。输入输出格式不一样，横向对比全靠手动对齐。现有工具面向 NeRF 或者闭源，前馈式模型没有对应的基础设施。而且模型能力停留在脚本级，下游应用想调用还得自己再包一层。

所以平台要解决三件事：统一部署让模型快速上线，统一对比让实验有可信度，API 封装让模型能力可以直接输出给下游。
```

### Slide 14 讲稿

```
平台的设计逻辑是覆盖从部署到输出的完整路径。

命令中心负责一键部署和任务提交。任务工作台全程追踪运行状态。样本矩阵把同一输入在不同模型下的结果并排展示。模型注册提供标准接口，新模型接入不需要改平台代码。能力输出把结果整理为统一格式，同时预留 API 接口——后续可以直接把模型推理能力暴露给外部应用调用。
```

### Slide 15 讲稿

```
技术实现上，平台靠"执行合同"这个核心抽象串起各层。

最上层是桌面前端，Tauri 2 加 React 加 TypeScript。第二层本地后端，FastAPI 负责模型注册、任务队列和合同验证。第三层远端调度，SSH 和 SCP 完成文件传输和进程管理。最底层是模型执行器，每个模型实现合同接口就能接入。

顶层预留了 API 导出接口——等模型验证通过，可以直接封装成服务供下游调用，不需要重新开发。

右边两张是平台当前运行截图。
```

### Slide 17 讲稿

```
平台后续方向分三步，对应三层价值。

短期是辅助实验——完善对比视图，保证消融和对照实验的数据链路完整。中期是快速部署——新模型接入后一键运行、即时对比，结果可直接导出用于论文。长期是 API 化——把验证通过的模型封装为标准接口，供 SLAM 初始化、AR 建模、机器人场景理解等应用直接调用。

从定位来看，平台从实验工具逐步演进为模型能力的输出通道。
```

### Slide 18 讲稿

```
回到课题本身。架构验证需要消融、对照和长序列评测，对条件一致性要求高。平台提供统一调度、统一格式、同条件对比和完整运行记录。

更进一步，架构验证通过之后，它本身可以作为新模型接入平台，走同样的标准流程。如果质量达标，还可以直接通过 API 层封装对外输出。所以平台的角色是：前期支撑验证，后期支撑输出。
```
