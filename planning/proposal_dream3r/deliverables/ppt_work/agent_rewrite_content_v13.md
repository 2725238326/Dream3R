### Slide 03 · 三维重建与 3R 模型的兴起

**页面内容：**
- 传统三维重建流程：特征提取 → 匹配 → 位姿估计 → 稠密重建（多步级联）
- 3R 方法的革新：输入图像对/序列，一次前向推理直接输出像素级三维点图
- 代表工作：DUSt3R（2024）开创，后续快速涌现十余个模型

**配图：** 左侧传统流程箭头图 vs 右侧 3R 单步示意图

**讲稿：**
传统三维重建需要多个步骤级联，流程长、容易累积误差。2024 年 DUSt3R 开创了前馈式 3R 方法，核心思路是把整个重建过程压缩到一次网络前向传播里，直接从图像得到三维点图。这个方向出来之后，一年多时间涌现了十几个后续工作。

### Slide 04 · 3R 模型的优势

**页面内容：**
- **速度快**：单次前向，无需迭代优化
- **端到端学习**：几何先验隐式学在网络权重里，泛化性好
- **统一输出表示**：点图（pointmap）天然支持多任务——深度、位姿、法线都能从中提取
- **各模型发展出的特色能力**：
  - 空间记忆（Spann3R）：用锚点存储已见区域
  - 流式递推（CUT3R）：逐帧递推处理任意长视频
  - 几何一致性校验（Test3R）：推理时做自检查
  - 跨视图注意力增强（MASt3R）：提升匹配精度

**讲稿：**
3R 模型有几个核心优势。第一是快，不需要像传统方法那样反复优化。第二是端到端学出来的几何先验泛化性好。第三是统一的点图输出格式——深度、位姿、法线都能算。而且各个后续工作还分别发展出了很有价值的特色机制，比如 Spann3R 的空间记忆、CUT3R 的流式递推、Test3R 的几何校验、MASt3R 的匹配增强。这些机制都是我们后面设计架构时重点学习的对象。

---

### Slide 05 · 3R 模型面临的共性缺陷

**页面内容：**
- **长序列漂移**：处理长视频时几何逐渐跑偏，百帧以上尤为严重
- **动态场景干扰**：移动物体导致静态假设失效
- **缺乏自检查**：推理结果没有置信度反馈，错了也不知道
- **单模型能力上限**：每个模型只擅长一类场景，弱纹理/镜面/快速运动等情况各有短板
- **模型之间孤立**：各自独立运行，没有统一框架来对比或组合

**讲稿：**
但 3R 模型也面临一些共性问题。长序列会漂移；碰到动态物体会被干扰；推理出来没有自检查机制，结果错了不知道；每个模型都只在特定场景下表现好，换个场景可能就不行了；而且模型之间各自独立，没有统一的框架来组合使用或者做公平对比。这些问题就是我们这个课题要针对性解决的。

### Slide 06 · 设计思路：从 3R 中学什么、用什么新方法补

**页面内容：**
- **从 3R 模型中学习的核心机制：**
  - 空间记忆思路 → 保留已见区域信息
  - 流式递推思路 → 支持任意长度输入
  - 几何校验思路 → 推理时做自检查
  - 多模型互补思路 → 多个专家各取所长

- **引入的新方法来弥补缺陷：**
  - 三分支稀疏注意力 → 高效管理长序列记忆
  - Mamba 状态空间模型 → 线性复杂度递推
  - Slot Attention → 动静物体分离
  - 能力描述符 + 路由策略 → 多专家按需调度
  - 几何信号融合（Sampson 残差 / 深度一致性 / 共视冲突）→ 自动检测错误

**讲稿：**
我们的设计思路分两步。第一步是从现有 3R 模型里提取值得学习的核心机制：空间记忆、流式递推、几何校验、多模型互补——这些都是被证明有效的方向。第二步是引入新方法来弥补它们的不足：用稀疏注意力和 Mamba 来高效管理长序列记忆，用 Slot Attention 处理动态物体，用能力描述符做多专家路由，用多类几何信号做自动错误检测。这两步合在一起，架构的方向就自然推出来了。

### Slide 09 · 空间记忆模块

**页面内容：**
- **要解决的问题**：长序列处理时全注意力 O(N²) 爆炸、信息遗忘
- **方案**：三分支稀疏注意力并行工作
  - 压缩分支：把历史帧压缩为固定长度状态向量（学自 CUT3R 递推思路）
  - 选择分支：空间锚点存储关键位置信息（学自 Spann3R 空间记忆）
  - 滑窗分支：Mamba + 局部 Transformer 混合（线性复杂度 + 局部精度）
- **效果**：三条路互补——压缩管远、选择管精、滑窗管近

**配图：** 三分支并行图 + 各分支输出汇聚

**讲稿：**
第一个核心是空间记忆模块。长序列处理最大的问题是注意力复杂度爆炸和信息遗忘。我们的方案是三条分支并行：压缩分支学 CUT3R 的递推思路把历史压缩成固定大小的状态；选择分支学 Spann3R 用空间锚点记住关键位置；滑窗分支用 Mamba 混合 Transformer 兼顾效率和局部精度。三条路各管一摊：压缩管远程记忆、选择管精确回忆、滑窗管当前上下文。

### Slide 10 · 几何校验模块

**页面内容：**
- **要解决的问题**：推理结果没有置信度反馈，错误无法被检测
- **方案**：三类几何信号联合检测 + 修复动作
  - 检测信号：Sampson 对极残差 / 相邻帧深度一致性 / 多帧共视冲突
  - 冲突评分超阈值 → 触发修复
  - 修复动作：不处理 / 局部帧重新推理 / 整窗口重跑 / 切换到其他专家
- **思路来源**：学习 Test3R 几何一致性校验的理念，但我们把校验做成架构内置组件而非独立后处理

**配图：** 三路信号 → 聚合 → 阈值 → 四种修复动作

**讲稿：**
第二个核心是几何校验模块。现有 3R 模型推理完就完了，没有自检查能力。我们的方案是把 Test3R 的校验理念学过来，但做成架构内置组件。具体是用三类几何信号——对极残差、深度一致性和共视冲突——联合打分，超过阈值就触发修复动作，轻则局部重跑，重则切换到另一个专家。这样架构自己就能发现并纠正错误。

### Slide 11 · 多专家编排模块

**页面内容：**
- **要解决的问题**：单个模型只擅长部分场景，碰到非优势场景性能下降
- **方案**：7 个 3R 模型作为专家池 + 路由策略
  - 每个专家标注能力描述符（擅长场景类型 + 速度等级）
  - 输入帧到来时，根据场景特征匹配最合适的专家
  - 校验模块检测到失败 → 自动切换到另一个专家重试
- **7 个专家**：MASt3R / Fast3R / Spann3R / CUT3R / MonST3R / MoGe-2 / DepthAnything-V2

**配图：** 七专家池 + 路由决策 + 校验联动箭头

**讲稿：**
第三个核心是多专家编排。既然单个模型不能包打天下，那我们就把 7 个模型都用起来，每个标注擅长什么场景，来了新输入就匹配最合适的专家。如果校验模块发现当前专家推理结果有问题，就自动切换到另一个专家重试。这样多个模型互补，整体鲁棒性比单打独斗要强。

### Slide 13 · 为什么需要一个聚合平台

**页面内容：**
- 现状问题：
  - 每个 3R 模型都有自己的运行环境、输入格式和输出格式
  - 跨模型对比需要手动配七套环境 + 手写转换脚本
  - 现有工具（Nerfstudio / 商业产品）要么面向 NeRF 不兼容，要么闭源不可审计
- 核心需求：
  - 多模型统一调度
  - 结果统一格式化方便对比
  - 支撑消融实验中的多专家对照

**讲稿：**
接下来讲软件平台。为什么要做这个平台？因为 7 个模型，七套环境七种格式，做一次对比实验要手动配七遍环境。Nerfstudio 是面向 NeRF 的跟 3R 不兼容，商业产品闭源不能审计算法。所以我们需要一个专门面向 3R 模型的统一管理平台，统一调度、统一格式、方便对比。

---

### Slide 14 · 平台核心功能

**页面内容：**
- **命令中心**：一键提交推理任务、查看所有模型状态
- **任务工作台**：实时追踪任务进度（上传→运行→回传→完成）
- **样本矩阵**：同一输入在不同模型下的结果并排展示
- **模型注册**：新模型只需编写一个执行器脚本即可接入
- **远程调度**：SSH/SCP 自动把任务推到 GPU 服务器执行
- **结果归集**：统一输出格式，支持导出对比报告

**配图：** 平台 UI 截图（待补）或功能框图

**讲稿：**
平台有几个核心功能。命令中心负责任务提交和状态总览；任务工作台追踪每个任务的实时进度；样本矩阵把同一组输入在不同模型下的结果并排展示方便对比；模型注册支持快速接入新模型；远程调度自动把任务推到 GPU 服务器；结果归集把各模型的异构输出统一起来。

### Slide 15 · 技术架构

**页面内容：**
- 四层设计：
  1. **桌面前端**：Tauri 2 + React + TypeScript（跨平台桌面应用）
  2. **本地后端**：FastAPI（模型注册、任务队列、状态管理）
  3. **远端调度层**：SSH/SCP（文件传输 + 进程管理 + 结果回传）
  4. **模型执行器层**：每个模型一个标准化执行器（封装差异）
- 核心抽象：统一执行合同（三个 JSON 文件：任务描述 / 状态 / 输出元数据）
- 代码规模：约 15000 行

**配图：** 四层架构图（从上到下）

**讲稿：**
技术架构是四层分离设计。最上面是 Tauri 加 React 的桌面应用负责界面，第二层是 FastAPI 后端管理任务队列，第三层通过 SSH 把任务推到远端 GPU 服务器，最底层每个模型有一个标准化执行器。核心抽象是三个 JSON 文件组成的统一执行合同——不管什么模型，提交、状态、输出格式都一样。整个平台大概 15000 行代码。

### Slide 18 · 平台如何支撑架构研究

**页面内容：**
- **消融实验调度**：多专家对照组通过平台统一提交，保证公平条件
- **结果可比性**：统一输出格式让不同模型结果可以直接数值对比
- **新架构接入**：我们的新架构训练完成后作为第 8 个模型接入，与现有 7 个做同条件对比
- **工程证据积累**：平台运行本身产生的调度数据、耗时对比也是课题成果的一部分

**配图：** 架构研究 ↔ 平台支撑的双向箭头图

**讲稿：**
平台和架构研究是互相配合的。架构的消融实验需要跑多个模型做对照，平台统一调度保证条件公平。结果格式统一了才能数值对比。等我们自己的新架构训练出来，直接作为第 8 个模型接入平台，和现有 7 个在同等条件下比较。所以平台是为架构研究服务的基础设施。

### Slide 19 · 实验设计

**页面内容：**
- **架构侧实验：**
  - 消融实验：逐一移除记忆分支 / 校验模块 / 多专家路由，观察性能变化
  - 长序列评测：窗口 10/20/50/100 帧，观察漂移程度
  - 多专家 vs 单一专家对照
  - 校验阈值标定：六类场景逐类标定
- **平台侧评测：**
  - 模型接入覆盖率（目标 7/7）
  - 跨模型对比矩阵完整性
  - 新模型接入边际成本
- 预估 GPU 约 1400 小时，按里程碑申请

**讲稿：**
实验设计分两块。架构侧主要做消融——把记忆、校验、多专家分别拿掉看性能变化多少；做长序列不同窗口长度的对比；做多专家组合和单一专家的对照。平台侧评估覆盖率、对比完整性和接入成本。GPU 预算约 1400 小时，按里程碑分批申请。

### Slide 20 · 创新点总结

**页面内容：**
1. **几何校验作为架构内置组件**：将推理时自检查机制从后处理提升为架构层模块
2. **异构多专家组合与路由**：7 个 3R 模型通过能力描述符按需调度
3. **三分支稀疏注意力记忆统一**：单一模块同时实现压缩/选择/滑窗三类长序列机制
4. **统一 3R 模型聚合管理平台**：标准化执行合同 + 四层架构 + 跨模型对比

**讲稿：**
创新点四个。第一是把几何校验做成架构内置的，推理时就能发现和修复错误。第二是多专家路由，让 7 个模型按场景特点各取所长。第三是三分支记忆设计，一个模块同时实现三种长序列处理机制。第四是聚合管理平台本身，统一合同加四层架构加跨模型对比能力。
