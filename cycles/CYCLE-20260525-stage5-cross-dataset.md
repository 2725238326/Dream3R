# CYCLE 20260525 Stage 5 Cross-Dataset (ETH3D)

## Scope

Activate the cross-dataset closure that DEC-20260525-005 deferred. The
KITTI-trained Stage 5 S1 expanded router (`router_stage5_s1_expand_v1`,
59-window KITTI, 78% LOO route accuracy) is evaluated zero-shot on ETH3D
Low-res many-view windows. Goal: probe whether the regime-stats routing
policy transfers to a second domain or whether the regime->expert mapping is
KITTI-specific.

No new training. No new checkpoints. The same three real experts
(Fast3R, MASt3R, Spann3R) used for KITTI S1 are evaluated on ETH3D windows.

The `_rig.7z` (distorted) and `_occlusion.7z` archives were not extracted

Per-scene observation coverage shows that only cam4 and cam5 have
triangulated COLMAP points (cam6 and cam7 have zero valid 3D point IDs in
`images.txt`); cam6/cam7 are dropped at dataset construction.

## Pointmap GT Construction

`code/dream3r/data/eth3d_long.py` defines `ETH3DLongSequenceDataset` which
mirrors `KITTILongSequenceDataset`'s output dict but builds patch-level
pointmap GT from COLMAP sparse SfM points:

1. Parse `cameras.txt` (PINHOLE: fx, fy, cx, cy per camera).
2. Parse `images.txt` (per-image world->camera pose QW QX QY QZ TX TY TZ
   + observations as `(u, v, POINT3D_ID)`).
3. Parse `points3D.txt` (POINT3D_ID -> X Y Z world-space).
4. For each window image: transform every visible 3D world point to camera
   frame via `p_cam = R(QW..QZ) @ p_world + (TX, TY, TZ)`; keep front-facing
   points with `p_cam.z > 0`.
5. Bucket points into `patch_grid x patch_grid` (14x14 = 196 patches) using
   the observed (u, v) pixel location relative to the camera's native
   width/height. Per-patch GT depth = median-z point in that bucket. Empty
   buckets get mask=0.

This matches the KITTI `pointmap_gt` shape `[N, 196, 3]` plus
`pointmap_mask [N, 196]`, so `build_oracle_expert_labels.py`'s
`_pointmap_abs_rel` runs unchanged.

Images are resized to 224x224 in the dataset; intrinsics scaled accordingly.
The COLMAP world->camera convention matches the KITTI long-window pose
convention (pose normalized relative to the first frame in each window).

## Regime Labels (Hardcoded Per Scene)

`code/dream3r/data/eth3d_regime_labels.json` is generated by
`code/dream3r/scripts/generate_eth3d_regime_labels.py`. The 6D regime
distribution is one-hot per scene (per HANDOFF design):

```text
delivery_area -> indoor_static
electro       -> indoor_static
forest        -> outdoor_static
playground    -> outdoor_static
terrains      -> outdoor_static
```

The 7 `STAT_FEATURE_KEYS` (frame_count, depth_mean, valid_ratio,
depth_temporal_change, oxts_available, mean_speed, speed_std) are computed
per-window from the dataset itself. ETH3D has no IMU + the rig is static,
so `oxts_available = mean_speed = speed_std = 0`. `depth_mean` comes from
the SfM-projected pointmap z-medians, `valid_ratio` from mask coverage.

The KITTI checkpoint's frozen normalization stats (stat_mean / stat_std)
apply directly via `_feature_tensor(..., frozen_stats=...)`, so the same
preprocessing pipeline that trained the router runs on ETH3D.

## Window Selection

Same per-window structure as KITTI long-window:

- 4 frames per window
- 224x224 image resize for expert forward
- 196 patches per frame
- Scale-aligned abs_rel metric (`--align-scale`)

ETH3D windows are sampled with `max_windows_per_scene=10` so the final
window set is balanced across scenes. After dropping cam6/cam7
(no SfM coverage), all valid windows come from cam5. Total: **50 windows
across 5 scenes** (10 per scene).

```text
KITTI 59-window: mast3r=31 (53%), spann3r=24 (41%), fast3r=4 (7%)
```

evidence that the per-window best expert depends on the domain.

## Cross-Dataset Transfer Eval

The Stage 5 S1 expanded router (`router_stage5_s1_expand_v1`,
`feature_mode=regime_stats`, n_train=59 on KITTI) is forwarded on ETH3D
windows without retraining. Frozen normalization stats are loaded from the
KITTI checkpoint.

Command:

```text
python -m dream3r.scripts.eval_cross_domain_router \
  --regime-labels /hdd3/kykt26/code/dream3r/runs/eth3d_cross_dataset_regime_labels/regime_labels.json \
  --oracle-labels /hdd3/kykt26/code/dream3r/runs/eth3d_cross_dataset_oracle/oracle_expert_labels.json \
  --router-checkpoint /hdd3/kykt26/checkpoints/router_stage5_s1_expand_v1/latest.pt \
  --kitti-oracle-labels /hdd3/kykt26/code/dream3r/runs/stage5_s1_expand_oracle/oracle_expert_labels.json \
  --output /hdd3/kykt26/code/dream3r/runs/eth3d_cross_dataset_router/results.json \
  --feature-mode regime_stats
```

Results:

```text

  random_routing: 0.2535683385
  oracle_router:  0.2074708661
  always_fast3r:  0.2712136716

  always_spann3r: 0.2324482446   <- best single on ETH3D
```

KITTI vs ETH3D cross-domain comparison (from `cross_domain` field):

```text

best_single_abs_rel       0.1608                0.2324

                                                {spann3r=23, mast3r=16, fast3r=11}
oracle_abs_rel            0.1482                0.2075
learned_router_abs_rel    0.1486 (closure) /    0.2712
                          0.1572 (LOO)
route_accuracy            96.6% closure /       22%
                          78%   LOO

                          2.23% LOO
```

Reading:

- The router does NOT transfer. It collapses to `always_fast3r` on every
  ETH3D window despite Fast3R being only the third-best single expert on
  ETH3D (oracle picks Fast3R for only 11/50 windows).
- Route accuracy 22% vs 33% chance is below random (and well below the
  46% "always-spann3r" majority-class baseline). The KITTI-learned regime
  -> expert mapping is actively wrong on ETH3D.
- The best-single-expert shifts (MASt3R on KITTI -> Spann3R on ETH3D), and
  the oracle distribution shifts (MASt3R-dominated on KITTI -> Spann3R-
  dominated on ETH3D). Both are real cross-dataset signals.

Why the router collapses (root cause):

- The `regime_stats` router input is `[6D regime probs] + [7D normalized
  stats]`. Stats are normalized with KITTI's frozen `stat_mean / stat_std`.
- KITTI training data has `oxts_available = 1.0` for every window; ETH3D
  has 0.0 (no IMU). KITTI's stat_std for oxts_available is ~1e-6 (constant
  across KITTI), so `(0 - 1.0) / 1e-6 ~= -1e6` — far outside the
  distribution the router ever saw at training time.
- Similarly KITTI's `mean_speed` has KITTI-only support (driving
  velocities ~10 m/s) which is 0 for the static ETH3D rig.
- The router's first MLP layer was never trained on these out-of-domain
  feature values; it produces a degenerate logit pattern that picks
  whichever expert dominates the corner-case region of the input space
  (Fast3R here).

## Limitations

- ETH3D Low-res many-view is a 5-scene SfM dataset. Sparse SfM points
  (~hundreds to thousands per image) are used as depth GT. Dense laser-scan
  depth GT (in `rig_scan_eval/scan*.ply`) is not consumed in this cycle.
- Only cam5 is used per scene (cam4 is also valid but not iterated under
  the current dataset order; cam6/cam7 have no SfM points). 50 windows
  total.
- The hardcoded one-hot regime probabilities for ETH3D scenes are scene-
  level (no per-window dynamics signal); this is the right design for
  static rig captures but means within-scene window-to-window routing
  variance comes from the stat features (depth_mean, valid_ratio,
  depth_temporal_change), not the regime probabilities.
- This is a zero-shot router transfer eval. No router retraining on ETH3D
  is performed in this cycle.
- Not SOTA. Not ScanNet. Not 4+ expert.

## Conclusion

The cross-dataset closure produces an honest negative finding: the
KITTI-trained Stage 5 S1 expanded router does NOT transfer to ETH3D. It
collapses to `always_fast3r`, gets 22% route accuracy (below 33% chance),
and is 16.7% worse than the ETH3D best single expert. The collapse is
explained by the stat normalization scheme baking in KITTI-specific feature
support (`oxts_available=1`, KITTI-driving `mean_speed` range) that ETH3D's
static rig cannot reproduce.

The closure does, however, surface two real cross-dataset signals worth
recording:

1. The per-window best expert *shifts* by domain (MASt3R on KITTI,
   Spann3R on ETH3D), confirming that the routing problem is non-trivial
   across datasets.
2. The oracle distribution changes from MASt3R-dominated on KITTI to
   Spann3R-dominated on ETH3D; Fast3R doubles its share (7% -> 22%).

Stage 5 closes with KITTI in-domain S1 as the headline (DEC-003) and this
cross-dataset evidence as a bounded, honest result. Cross-domain routing
is left as future work in DEC-20260525-006.
