# Dream3R PPT Formal Presentation Prompt

Status: active handoff prompt for formal meeting-style presentation.

Date: 2026-05-11

## Purpose

Use this prompt when asking another agent to create a PPT from the Dream
workspace. The PPT is for a teacher or research discussion, not for an
internal implementation handoff. It should translate the technical work into
a clear project narrative without promotional language.

## Prompt

```text
请基于 E:\kykt\Dream 下的工作文档制作一份中文研究汇报 PPT，用于和老师或合作者开会讨论。请注意：这不是内部代码说明，也不是宣传材料。不要照搬工作文档中的大量英文模块名、测试命令和 agent 术语。请采用正式组会汇报的写法，重点讲清楚研究动机、方案思路、已完成工作、当前证据边界、后续计划和应用落地路径。

参考文档：
- E:\kykt\Dream\TASK_SNAPSHOT.md
- E:\kykt\Dream\README.md
- E:\kykt\Dream\WORKFLOW_STATUS.md
- E:\kykt\Dream\RESEARCH_STATE.md
- E:\kykt\Dream\code\dream3r\RECENT_PROGRESS.md
- E:\kykt\Dream\code\dream3r\DEMO_SUMMARY.md
- E:\kykt\Dream\code\dream3r\REAL_DATA_SMOKE.md
- E:\kykt\Dream\code\dream3r\SOTA_FEATURE_MATRIX.md
- E:\kykt\Dream\code\dream3r\RESEARCH_BASE_AND_INNOVATIONS.md
- E:\kykt\Dream\code\dream3r\NEXT_PHASE_ROADMAP.md

整体要求：
1. 中文为主。英文术语只在必要时保留，并用括号解释。
2. 语气要像正式组会汇报，避免口号式、宣传式和 AI 生成感很强的句子。
3. 不要使用“颠覆”“突破性”“一眼看懂”“空间智能系统”等泛化宣传词。
4. 不要写“金句”。总结页用正常研究汇报语言。
5. 每页 3-5 个要点，少用长段落。
6. 明确区分“已经完成”“已经验证”“尚未证明”“下一步计划”。
7. 不要声称已经达到 SOTA，也不要暗示已有论文级实验结果。
8. 技术细节放在备份页，主讲页只保留必要概念。
9. 图示优先使用中文标签，例如“感知模块”“空间记忆”“几何自检”“多模型调度”“结果展示”。

建议标题：
面向长序列场景的三维重建方法研究

副标题：
阶段性工作汇报：系统方案、原型验证与后续实验计划

建议页数：10-12 页。

第 1 页：标题页
- 标题：面向长序列场景的三维重建方法研究
- 副标题：阶段性工作汇报：系统方案、原型验证与后续实验计划
- 可在页脚标注内部原型名：Dream3R

第 2 页：研究背景和问题
- 三维重建从单次输入走向长序列、多视角和动态场景时，会出现漂移、遗忘和错误累积。
- 现有方法各有优势，但通常难以同时兼顾精度、效率、长期状态维护和错误诊断。
- 我们关注的问题是：如何让重建过程具备连续处理、状态保留和结果检查能力。

第 3 页：总体思路
- 不把目标限定为复现某一个模型，而是先建立可扩展的重建流程。
- 将感知、空间记忆、几何检查、动态对象处理和多模型调用拆成相对独立的环节。
- 每个环节保留可记录的中间信号，便于后续调试、消融和展示。
- 内部原型命名为 Dream3R。

第 4 页：系统结构
请画一张中文结构图：
图像/视频输入 -> 感知与特征提取 -> 空间记忆 -> 几何自检 -> 多模型调度 -> 重建结果与可视化输出
旁边标注：
- 长期空间记忆
- 动态对象状态
- 错误检查与修正建议
- 多模型能力接口

第 5 页：前期研究和文档工作
- 建立了研究流程和决策记录，避免直接进入无约束复现。
- 梳理并比较了多条 3R 相关技术路线。
- 形成了架构设计、实验计划、风险边界和后续路线图。
- 保留证据标签，区分论文已知结论、工程验证和推测性判断。

第 6 页：代码原型进展
- 已实现第一版可运行原型。
- 支持连续窗口输入、空间记忆、几何自检、多模型接口和状态更新。
- 已加入基础训练、评估、消融和可视化导出入口。
- 服务器端完成同步和测试验证。

第 7 页：当前验证结果
- 服务器 smoke test 通过。
- 主要单元测试通过。
- 合成数据上可以观察状态更新、空间记忆写入和几何自检信号。
- KITTI 真实 RGB/depth 序列已完成两窗口初步运行。
- 当前真实数据结果用于验证流程可运行，不代表最终重建质量。

第 8 页：证据边界
做成两栏：
左栏“目前可以说明”：
- 原型已从方案推进到可运行代码。
- 真实数据可以进入完整处理流程。
- 系统能够记录重建、记忆、自检和修正相关指标。

右栏“目前还不能说明”：
- 不能说明重建质量已经超过现有方法。
- 还没有完整真实数据消融实验。
- 自检阈值和多模型调度效果还需要进一步评估。
- 三维可视化渲染仍处于后续接入阶段。

第 9 页：应用落地考虑
- 输出深度、点云、置信度等基础重建结果。
- 累积稳定空间信息，用于长序列场景表达。
- 标记可能存在冲突或不稳定的区域，便于人工检查。
- 后续接入三维可视化，形成可查看的场景结果。
- 可与 KYKT 中的重建结果展示、模型对比和诊断报告结合。

第 10 页：后续实验计划
- 在真实数据上做消融实验，比较不同状态更新、空间记忆和自检设置。
- 校准几何自检模块，减少误报和漏报。
- 评估多模型调度是否带来质量或效率收益。
- 完善 DTU / KITTI 数据路径和指标统计。
- 在依赖条件明确后推进三维渲染输出。

第 11 页：风险和边界
- 合成数据演示不能替代真实数据实验。
- 真实重建质量需要训练、校准和系统消融支撑。
- 部分外部模型和渲染依赖仍需单独处理。
- 后续展示必须区分“流程跑通”和“性能提升”。

第 12 页：总结
- 本阶段完成了从研究路线、方案设计到代码原型的推进。
- 当前原型已经具备连续输入处理、空间记忆和几何自检等关键功能。
- 真实数据初步流程已经跑通，但性能结论仍需后续实验支持。
- 下一阶段重点是补齐真实数据消融、质量评估和结果可视化。

备份页：术语对照表
- Dream3R：内部原型名
- MemoryBus：模块间信息传递机制
- AnchorBank：长期空间记忆库
- Critic：几何自检模块
- ComposerRouter：多模型调度模块
- NSA：稀疏记忆检索机制
- Mamba hybrid：时序状态更新机制
- GaussianHead：面向三维可视化的结果表示接口
```
