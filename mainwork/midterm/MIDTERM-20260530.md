# Dream3R Midterm Report — Architecture State

**Report date**: 2026-05-30
**Period covered**: 2026-05-23 (Stage 1 kickoff) → 2026-05-29 (DEC-009 closure)
**Author**: buaabw

---

## §0 TL;DR

Dream3R is a 5+ stage closed-loop 3D reconstruction pipeline. As of 2026-05-29:

- **Stages 1-3 closed**: DINOv3 backbone + 3 real expert adapters (fast3r,
  mast3r, spann3r) + SpatialMemory (anchor bank + NSA) + ComposerRouter,
  all running end-to-end on real KITTI + ETH3D imagery.
- **Stage 4 partial**: Critic trained on 24-example supervised dataset; the
  reroute closed loop into Stage 5 was tested and closes **negative** at
  current data scale (DEC-008).
- **Stage 5 strong**: 3-expert routing on KITTI 59w + ETH3D 50w with a
  joint domain-aware router (joint v3 dense). 5-seed LOO sweep shows
  **+6.15% ± 0.65pp ETH3D rel_imp** (all 5 seeds positive), KITTI rel_imp
  near zero and sign-unstable (DEC-007 addendum 3).
- **Structural finding (this week)**: V04Pipeline's final depth output =
  one selected expert's standalone pointmap. Anchor bank / NSA / memory /
  critic state populate `ReconstructionOutput` but are **never consumed by
  anything downstream of expert dispatch**. The architecture's depth-side
  value-add is fully captured by routing quality.
- **Architecture pivot (new this week)**: hard expert selection is now
  treated as a diagnostic baseline / proposal prior, not the final
  Dream3R contribution. The post-midterm architecture is
  **state-conditioned reconstruction**: persistent Memory / Anchor / NSA
  state and Critic/geometric reliability must directly condition the final
  pointmap (§3.5, DEC-009).

The midterm question — *does the architecture work?* — has a layered answer:

1. The **routing-side architecture** (Stages 1-3 + Stage 5) is validated:
   +3-6% LOO rel_imp vs best single expert, 5-seed seed-robust on ETH3D.
2. The **closed-loop Critic→router** (Stage 4 + Stage 5) does **not**
   transfer at current data scale (24-example training, 109-window eval).
3. The **hard-selection architecture** has a clear ceiling: final depth is
   one selected expert's standalone output.
4. The next Dream3R contribution is **state-conditioned reconstruction**:
   use long-sequence state and reliability signals to fuse/correct the
   final pointmap, not merely to pick an expert.

## §1 Architecture overview

### 1.1 Stage diagram

```text
        ┌──────────────────────────────────────────────────────────┐
        │  Stage 1: Perception (DINOv3 ViT-B/16 ONNX, frozen)      │
        │    images [B,N,3,H,W]  →  feature_tokens [B,N,P,D]       │
        │                          pointmap_proposal [B,N,P,3]     │
        └──────────────────────────────────────────────────────────┘
                              ↓
        ┌──────────────────────────────────────────────────────────┐
        │  Stage 2: SpatialMemory (AnchorBank + NSA attention)     │
        │    cross-window state + 17 named evidence channels       │
        │    → latent_drift_proxy, bank_occupancy, fused_context   │
        └──────────────────────────────────────────────────────────┘
                              ↓
        ┌──────────────────────────────────────────────────────────┐
        │  Stage 3 / Stage 5: Composer (ComposerRouter)            │
        │    12D input = 6 regime probs + 4 robust stats +         │
        │                2 domain one-hot                          │
        │    → selected_expert, routing_logits, route_recommendation│
        └──────────────────────────────────────────────────────────┘
                              ↓
        ┌──────────────────────────────────────────────────────────┐
        │  Stage 4: Critic + RepairExecutor                        │
        │    evidence + pointmap_pair + confidence_pair            │
        │    → conflict_score, repair_action (0..4)                │
        │    Action 3 = reroute to route_recommendation[:,1]       │
        └──────────────────────────────────────────────────────────┘
                              ↓
        ┌──────────────────────────────────────────────────────────┐
        │  Expert dispatch: composer.dispatch(expert_id, images)   │
        │    Real adapters: fast3r / mast3r / spann3r              │
        │    → expert.pointmap [B,N,P,3]  (← FINAL depth output)   │
        └──────────────────────────────────────────────────────────┘
```

  `code/dream3r/data/` (extensions and new data builders only).

### 1.3 Cross-stage data contracts (`code/dream3r/contracts.py`)

Each stage publishes a typed dataclass. `ReconstructionOutput` is the
final pipeline output and carries all upstream state:

```python
ReconstructionOutput(
    pointmap,            # [B,N,P,3] — FINAL depth, = expert.pointmap
    confidence,
    perception,          # PerceptionOutput
    memory,              # MemoryOutput  ← carries anchor/NSA state
    permanence,
    critic,              # CriticDecision ← carries conflict_score
    composer,            # ComposerDecision ← carries route_recommendation
    expert,              # DispatchedExpertOutput ← carries pointmap
    repair,              # RepairReport
    ...
)
```

This contract is the key handle for Stage 6 (§4): all of `memory`,
`critic`, `expert` are available **but only `expert.pointmap` becomes the
final `pointmap`**. The other fields are emitted but consumed only by
logging/observability.

> The **KITTI-side joint rel_imp is essentially zero on average and
> sign-unstable** across seeds (+0.87% ± 1.62pp, 4/5 positive, 1/5
> negative). KITTI router still does not zero-shot transfer to ETH3D
> (route_acc 42%, rel_imp -22.59% under dense GT).

### 2.5 Stage 4 — Critic closure + Stage 4↔5 closed-loop (DEC-008)

The Critic was trained on 24 examples (12 KITTI sequences × 2 experts;
fast3r/mast3r only, spann3r absent). Stage 4 closure passed in isolation
(conflict↔abs_rel correlation 0.834 on the training set). Wiring into
Stage 5's 109-window cross-domain set fails:

| metric | gate | KITTI | ETH3D |
| --- | --- | --- | --- |
| conflict_score_sigmoid std | > 0.1 | **0.017** ✗ | **0.038** ✗ |
| conflict-based expert-pick acc | > 33% chance | **25.4%** ✗ | 25%-ish ✗ |
| `recommended_action == reroute_model` count | > 0 | **0 / 177** | **0 / 150** |

The trained Critic does not transfer beyond its 24-example training set.
Its conflict head saturates near 0.55 sigmoid; it never recommends
reroute on the broader 109-window evaluation set.

Critic's **deterministic geometric features** (sampson_distance,
depth_inconsistency, both computed without learned parameters) ARE
informative — sampson picks the best expert on 52% (KITTI) / 42% (ETH3D)
of windows, above 33% chance. However, used as an inference-time
reroute heuristic, they degrade aggregate LOO performance:

| LOO (5 seeds, sampson m=0.10) | router-only | with reroute | delta |
| --- | --- | --- | --- |
| KITTI rel_imp mean | +2.16% ± 1.33 | +1.41% ± 1.01 | **-0.75pp** |
| ETH3D rel_imp mean | +9.11% ± 1.64 | +6.99% ± 2.15 | **-2.12pp** |

**Plan criterion 4 FAILS** — no domain improved by ≥1pp on average.
Reading: the geometric features carry partial signal (above chance) but
lower per-window accuracy than the router (LOO 70-75%); reroute
overrides correct router picks more often than it fixes incorrect ones.

DEC-008 outcome category: **NEGATIVE, valid closure** (matches DEC-007
honesty norms).

## §3 Structural finding (this week)

### 3.1 The depth-output mechanic

Reading `code/dream3r/orchestrator.py:530`:

```python
return ReconstructionOutput(
    pointmap=expert.pointmap,    # <— final depth = expert's standalone output
    confidence=expert.confidence,
    ...
)
```

`expert.pointmap` is the result of
`composer.dispatch(expert_id, images, context={"regime_probs": ...,
"critic_confidence": ...})`. The `context` carries only regime probs and
critic conflict score; **`memory.fused_context`, NSA branch weights,
anchor bank state, permanence dynamic mask are NOT passed to dispatch**.

The expert adapter (`Fast3RAdapter` / `MASt3RAdapter` / `Spann3RAdapter`)
runs its standard `forward(images)` — same call that
`build_oracle_expert_labels.py` uses to populate the oracle metric
tables. Therefore:

```text
pipeline_abs_rel[seq] = oracle_table[seq][composer.selected_expert[seq]]
```

V04Pipeline's depth output, on any window, equals one of the three
standalone expert pointmaps that already live in the oracle tables. The
architecture's contribution to depth abs_rel is fully expressed by the
routing decision (which expert is picked).

### 3.2 Implications for past experiments

This explains why several DEC-008-adjacent experiments collapsed:

- **Critic-augmented joint router training** (DEC-008 §Phase B planned):
  would have trained a router on a signal (`conflict_score_sigmoid`)
  that didn't transfer beyond 24 examples (Finding #1 above).
- **Geometric reroute closure-set sweep** (DEC-008 Phase B actual):
  reroute fires inside V04Pipeline-equivalent logic, so its effect = the
  difference between (router-picked expert)'s standalone abs_rel and
  (top-2 expert)'s standalone abs_rel. Same conclusion as eval_router_*
  scripts; no new information.
- **End-to-end pipeline vs single-best comparison** (proposed but not
  run): would have computed `mean(oracle_table[seq][router_pick[seq]])`
  for V04Pipeline and compared against `mean(oracle_table[seq][best_single])`
  — already in DEC-007 as "router rel_imp vs best_single".

### 3.3 Architecture's design intent vs current implementation coverage

| Stage | Design intent (STAGE-*.md) | Current implementation |
| --- | --- | --- |
| Perception | DINOv3 features → downstream | ✓ wired |
| Memory (anchor + NSA) | Long-sequence drift reduction via persistent state | ✓ trained; state ✓ emitted to `memory_log` / `memory.fused_context`; ✗ **NOT consumed for depth** |
| Permanence | Dynamic mask, CR-2 write gating | ✓ emitted; ✗ not consumed for depth |
| Composer (router) | Multi-expert routing | ✓ trained; ✓ consumed (picks expert) |
| Critic | Conflict detection + repair | ✓ emitted; partially consumed (reroute is wired but never fires per DEC-008) |
| Expert dispatch | Real adapter forward | ✓ produces final depth |

**The vertical wire from Memory / Permanence / Critic state into the
depth-producing path is missing in the current implementation.** This is
not a bug — it matches the v0.4 orchestrator design where the expert is
treated as a black-box adapter. But it bounds what any experiment using
abs_rel as the metric can reveal about Memory / Permanence / Critic.

### 3.4 What this changes about the project narrative

The "5-stage composable closed-loop architecture" story we have been
telling is half-implemented:

- The **routing-side closed loop** (Critic emits → Composer reads via
  `critic_confidence` context kwarg + `reroute_hint`) is wired and
  testable, but inert in practice (DEC-008 Finding #1).
- The **state-to-depth wire** (Memory state → refined pointmap) is **not
  in the orchestrator**; it has to be added explicitly.

DEC-007 and DEC-008 cover the parts that ARE wired. To advance the
architecture **beyond what those closures already say**, we have to add
the missing wire — which is what Stage 6 (next section) attempts as the
smallest possible architectural addition.

### 3.5 Route adjustment: from expert selection to state-conditioned reconstruction

DEC-009 (`DEC-20260527-009-state-conditioned-reconstruction-pivot.md`)
formalizes the architecture adjustment.

The old v0.2 headline:

```text
Heterogeneous best-of-N Composer
```

is demoted to:

```text
proposal prior / regime probe / diagnostic baseline
```

The new post-midterm headline is:

```text
state-conditioned reconstruction
```

Concretely, Dream3R should move from:

```text
ComposerRouter picks one expert
-> expert.pointmap is final pointmap
```

to:

```text
fast3r / mast3r / spann3r proposal pointmaps
+ proposal confidences
+ memory.fused_context
+ anchor stability / NSA state
+ critic conflict + geometric reliability
+ composer prior
-> fused / corrected final pointmap
```

This preserves DEC-007's value: routing experiments prove that regime and
domain signals are learnable. It also preserves DEC-008's negative result:
reroute is not enough. The architecture change is the synthesis of both:
Composer should schedule and weight proposals, while Memory/Critic state
must enter the reconstruction plane directly.

## §4 Stage 6 — Post-dispatch fusion head prototype (new this week)

### 4.1 Design

**Goal**: connect `memory.fused_context` and `critic.conflict_score` to
the final pointmap output, without modifying v0.3/v0.5 core.

**New module**: `code/dream3r/fusion_head.py` — `Stage6FusionHead`:

```text
inputs:
    expert_pointmap    [B, N, P, 3]    from DispatchedExpertOutput
    expert_confidence  [B, N, P, 1]    from DispatchedExpertOutput
    memory_context     [B, D]          from MemoryOutput.fused_context
    conflict_score     [B, 1]          from CriticDecision.conflict_score

architecture:
    1. project memory_context [B,D] → [B, head_dim] via nn.Linear
    2. broadcast to per-patch: [B, head_dim] → [B, N, P, head_dim]
    3. concat with pointmap + confidence + sigmoid(conflict):
       [B, N, P, 3 + 1 + head_dim + 1]
    4. small MLP (3 layers, hidden=128) → delta [B, N, P, 3]
    5. refined = expert_pointmap + tanh(conflict_score) * delta
       (identity when critic is confident; refinement only when conflict high)

trainable params: O(50k-100k)
```

### 4.2 Subclass extension pattern (no v0.3 core edits)

**New script**: `code/dream3r/scripts/v04_pipeline_with_fusion.py` —
`V04PipelineWithFusion(V04Pipeline)` subclass:

```text
overrides:
    forward(...) -> ReconstructionOutput:
        out = super().forward(...)  # full V04Pipeline pass
        refined = self.fusion_head(
            out.expert.pointmap,
            out.expert.confidence,
            out.memory.fused_context,
            out.critic.conflict_score,
        )
        return dataclass.replace(out, pointmap=refined)
```

`code/dream3r/orchestrator.py` itself is **not modified**; the subclass
adds Stage 6 by extension only. This preserves the CLAUDE.md hard
constraint on v0.3 core immutability.

### 4.3 Training setup

**New script**: `code/dream3r/scripts/train_fusion_head.py` — mirrors
`train_critic_only.py` structure:

1. **Cache build pass** (~30-60 min on GPU 1): one `V04Pipeline.forward()`
   per window over KITTI 59w + ETH3D 50w; saves
   `(expert_pointmap, expert_confidence, memory_context, conflict_score,
   gt_pointmap, gt_mask)` per window to `runs/stage6_fusion/{kitti,eth3d}_cache.pt`.
2. **Training loop**: 200-500 epochs, Adam(lr=1e-3), batch over windows;
   loss = `_pointmap_abs_rel(refined, gt, mask, align_scale=True)` (same
   metric as oracle).
3. **Eval split**: 5-seed held-out (80/20) per domain; mirrors
   `run_geometric_reroute_sweep.sh` pattern.

### 4.4 Preliminary results (Day 2 PM) — invalidated by baseline pathology

5-seed sweep launched on server 2026-05-27 13:28; 3/5 seeds completed
by 13:44. Head training converges (train loss drops 26-95% across
seeds, 300 epochs). Per-seed numbers:

| seed | KITTI baseline | KITTI head | KITTI Δ | ETH3D baseline | ETH3D head | ETH3D Δ |
| --- | --- | --- | --- | --- | --- | --- |
| 7  | 0.9366 | 0.3356 | +64.2pp | 0.5282 | 0.4043 | +23.5pp |
| 11 | 0.9323 | 0.3511 | +62.3pp | 0.5376 | 0.4945 | +8.0pp |
| 13 | 1.0014 | 0.3873 | +61.3pp | 0.4971 | 0.4123 | +17.1pp |

**Pathology**: the KITTI baseline abs_rel of 0.93-1.00 is ~6× worse
than DEC-007's standalone fast3r/mast3r/spann3r baseline (~0.14-0.16).
Investigation of the cache (`runs/stage6_fusion/kitti_cache.pt`):

```text
expert_name distribution: Counter({'fast3r': 246})
```

The model used here is `build_dream3r("small_real")`, which calls
`composer.load_from_registry()` (registers expert names) but **never
calls `adapter.load_checkpoint()`** on the individual adapters. When
V04Pipeline's `_dispatch_expert` runs `self.model.composer.dispatch(...)`,
fast3r returns its **fallback deterministic stub output**, not real
fast3r forward. Baseline abs_rel reflects this stub, not the real
expert.

**Implication**: Stage 6 head learned to compensate for the fallback
stub, not to refine a real expert's depth. The +60pp KITTI delta is
real-but-meaningless — it's the head finding a better depth
representation than the random-init stub, which is trivially possible
(any reasonable function beats noise). This does NOT show that anchor
state contributes refinement beyond a real expert.

The fix is to plumb `adapter.load_checkpoint()` into the V04Pipeline
cache-build path, matching what `build_oracle_expert_labels.py:91-99`
does. With loaded adapters, baseline abs_rel will drop to ~0.14-0.16
(matching the oracle), and the head's room to improve will be much
smaller — the actual null/positive/negative outcome from §4.4 will
become measurable.

**This rerun is the next concrete task** (see handoff
`mainwork/HANDOFF-20260527-stage6.md`).

## §5 Post-midterm roadmap

The roadmap changes from "make the router stronger" to "make state
condition the reconstruction output."

### 5.1 L0 — real-backend guardrail

Before any new architecture result is interpreted, real expert baselines
must be real:

- cache builders that claim `small_real` must call `adapter.load_checkpoint()`
  or assert `adapter.is_loaded`;
- cache entries should record per-window backend status;
- fallback-stub baselines must fail fast rather than enter result tables.

This directly addresses the Stage 6 baseline pathology in §4.4.

### 5.2 L1 — single-expert state-to-depth wire

Keep the current Stage 6 head as the smallest valid probe:

```text
selected expert pointmap
+ expert confidence
+ memory.fused_context
+ critic conflict_score
-> residual correction
```

Its verdict should be interpreted as:

- positive: current Memory/Critic state already helps depth correction;
- null: the wire is correct, but the state is not yet depth-informative;
- negative: naive residual correction damages strong expert outputs.

### 5.3 L2 — multi-expert proposal-bank fusion

Move beyond hard expert selection:

```text
all expert pointmaps/confidences
+ composer prior
+ state/reliability signals
-> proposal weights + residual correction + final pointmap
```

This is the first architecture that can claim more than "we picked the
right expert." Composer remains useful as a cost-aware scheduler and prior,
but the final output is fused/corrected geometry.

### 5.4 L3 — long-sequence objectives

Abs_rel stays as a sanity metric, but the Memory/Anchor/NSA claim needs
sequence metrics:

- scale drift proxy;
- temporal point stability;
- cross-window consistency;
- anchor reuse / survival;
- conflict reduction before/after correction.

These metrics align evaluation with Dream3R's actual intended advantage:
long-context state, not just per-window expert quality.

### 5.5 L4 — state representation retraining

If L1/L2 are null or weak, the next bottleneck is probably the state
representation itself. Then the follow-up is to train Memory/Anchor state
with depth/coherence supervision. This may require v0.3/v0.5 core
exemptions and should be gated by a separate DEC.
