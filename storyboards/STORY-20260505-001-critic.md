## Narrative Beats

ordered_steps:

```text
1. start state: a hard pair arrives. Two candidate models exist;
   the system has not yet picked.
2. trigger: Model A is run first (default route). Output looks
   plausible but one region is geometrically inconsistent.
3. action: Critic verifies the pointmap. Conflict signal fires on
   the inconsistent region.
4. action: Critic proposes reroute_model from Model A to Model B.
   Composer is asked: "do you agree on the capability spread?"
5. action: Composer publishes capability_match. Spread is nonzero
   and Model B wins.
6. visible result: Critic emits A5 reroute_model. Model B's output
   is shown. The inconsistent region is gone.
7. takeaway frame: "Without the verify-and-reroute step, the bad
   Model A output would have shipped. The Critic catches it; the
   Composer agreement keeps the catch from being a heuristic."
```

surprise_hook:

```text
"Catch a near-failure that other 3R systems silently accept, and
repair it on the spot."
```

This hook is locked. Other candidate hooks considered and retired in this draft:

- "See the geometry conflict scoreboard" — retired: too internal-mechanism; audience profile is cold-start, scoreboard reading requires comparator vocabulary.
- "Two models disagree, Dream picks the right one" — retired: framing reduces the contribution to "model selection" (Composer territory) rather than "verify-then-act" (Critic contribution); also collapses the cross-spec contract story.
- "Test3R-style verify, but with reroute" — retired: presupposes Test3R awareness, which the audience profile says is not assumed.

The locked hook survives the audience-profile filter (cold-start + 创新范式 + 讲好故事) and survives the "this is just paper X" collapse risk (per DEC-20260505-001 (1) rationale, the locked hook makes the *repair* action — not the verify action — the protagonist).

## What Could Go Wrong On The Day

failure_modes_during_demo:

- comparator output looks identical to Dream output (no visible difference)
- chosen sample turns out to be too easy (no conflict / no dynamics / no regime distinction)
- annotation labels look subjective when shown
- offline post-processing is mistaken for live inference
- Q&A surfaces a comparator the spec did not address

per_failure mitigations:

```text
1. comparator output identical: pre-select the demo pair from
   CRITIC-02's Fast3R-vs-Spann3R regime where capability_match spread
   is paper-derived as nonzero. If on the day the chosen pair turns
   out to look identical anyway, fall back per "Fail-Safe Alternative"
   below.
2. sample too easy: pre-vet the sample with the cycle-009 CRITIC-02
   `comparator_3 Test3R-style verify-then-report` row — if even a
   verify-only comparator has 0 expected detection on this sample,
   retire the sample. Use CRITIC-02's regime as the canonical hard
   pair.
3. annotation subjectivity: avoid showing pointmap_conflict
   thresholds; show only the binary "Critic flagged this region"
   indicator. Quantitative thresholds stay in the case-card text.
4. offline mistaken for live: every panel labeled "schematic" or
   "paper-derived" on its face per the functional_vs_placeholder
   boundary above. The opening slide states "panels are paper-derived
   schematics; no live inference today."
5. Q&A on unaddressed comparator: redirect to `cases/CASE-20260504-
   CRITIC-01..03.md` "Comparator Policies" tables. If the comparator
   genuinely is unaddressed (e.g. CUT3R, an OVGGT variant), label
   the answer "out of scope of this card; recorded for cycle 012";
   do NOT improvise a comparator analysis on the spot (Discipline
   rule 5 Honesty Override).
```

If any failure mode lacks a written mitigation, the storyboard is retired, not shown.
